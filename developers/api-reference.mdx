## Overview

EzyText exposes a REST API for your workspace's resources — contacts, campaigns,
segments, polls, inbox, and more. Authenticate with an
[API key](/developers/api-keys) and call the same resources you use in the app.

<Info>
  **Difficulty:** Advanced
</Info>

## Authentication

Create an [API key](/developers/api-keys), then send it on every request:

```http
Authorization: Bearer et_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

or

```http
X-API-Key: et_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

An API key authenticates with **Owner-level** access to your workspace.

## Interactive docs

The full, always-current endpoint reference (request/response schemas) is served as
interactive API docs by the platform. Ask your administrator for your instance's
API docs URL (typically `/docs`), and the raw OpenAPI spec (typically `/docs-json`)
if you want to generate a client.

<Note>
  API keys **cannot** manage other API keys or webhooks — those endpoints require a
  signed-in user. Everything else you can do in the app is available to a key with
  Owner access.
</Note>

## Example

```bash
curl https://<your-api-host>/contacts \
  -H "Authorization: Bearer et_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
```

## FAQ

<AccordionGroup>
  <Accordion title="Is the API versioned?">
    The REST paths aren't version-prefixed today. Follow the interactive OpenAPI spec
    for the current contract.
  </Accordion>
  <Accordion title="Which resources can I access?">
    The same org-scoped resources you use in the app — contacts, campaigns, segments,
    polls, inbox, and more.
  </Accordion>
  <Accordion title="How do I get notified of events?">
    Use [webhooks](/developers/webhooks) rather than polling the API.
  </Accordion>
</AccordionGroup>
